# Roadmap: py-tg-moder

Цель: превратить бота из набора разрозненных хендлеров в управляемого модератора одного чата с реальной фильтрацией контента и полноценным инструментарием админа.

**Статус: все три фазы реализованы и смерджены в `main`.** Дальнейшие идеи собраны в разделе [«Что дальше»](#что-дальше).

Оценка трудозатрат: **S** ≤ полдня, **M** ≤ 2 дней, **L** — больше.

## Фаза 0 — Хардненинг и быстрые правки ✅

Дешёвые изменения, закрывающие явные баги и небезопасное поведение. Делаются первыми, друг от друга почти не зависят.

### 0.1 Allowlist чатов + задействовать `config.yaml` `[M]` ✅

Сейчас `config.yaml` загружается, но `cfg` не используется нигде — бот работает в любом чате, куда добавлен.

- Превратить `main_group` в список `allowed_chats` (id или @username).
- Резолв @username → id один раз на старте; хранить `set[int]`.
- Декоратор/гард `@restricted_to_allowed_chats`, оборачивающий все хендлеры; чужой чат → бот молча выходит (`leave_chat`) или игнорирует.

**Готово:** в неразрешённом чате бот не реагирует ни на одну команду/событие; конфиг — единственный источник списка.

### 0.2 Починить `/unmute` `[S]` ✅

`UNMUTE_PERMISSIONS` не перечисляет `can_send_polls`/`can_change_info`/`can_pin_messages` → после размьюта опросы остаются запрещены.

- Привести набор полей к симметрии с `MUTE_PERMISSIONS` (все `True`, кроме осознанно закрытых).
- Тест на полноту полей обоих объектов (помогает не разъезжаться при обновлении PTB).

**Готово:** после `/unmute` пользователь восстановлен в полных правах чата.

### 0.3 Обратная связь и зачистка команд `[M]` ✅

Команды модерации сейчас пишут только в лог сервера; сообщение `/ban` остаётся в чате.

- На успех — короткая реакция в чате (реплай «🔨 забанен» / реакция-эмодзи) или удаление команды.
- На отказ прав у бота (нет admin) — внятное сообщение админу + лог в Sentry.
- Удалять само командное сообщение после обработки.

**Готово:** по любому исходу команды видно понятный результат, мусор-команды не копятся.

### 0.4 `/ban` удаляет спам-сообщение `[S]` ✅

- При бане по reply — удалять и сообщение, на которое отвечают.
- Сделать поведение настраиваемым (`delete_on_ban: true`) на будущее.

**Готово:** бан по reply убирает и нарушителя, и его сообщение.

### 0.5 Защита целей `[S]` ✅

- Перед действием проверять, что цель — не админ/владелец/сам бот; иначе дружелюбный отказ.

**Готово:** попытка забанить админа/бота даёт понятное сообщение, а не ошибку в Sentry.

### 0.6 `/help` и `/start` `[S]` ✅

- `/help` — список команд и правил поведения бота; для админов — расширенный.
- `/start` в личке — краткое «я приватный бот чата X».

**Готово:** команды задокументированы внутри самого бота.

## Фаза 1 — Ядро модерации (главный продуктовый пробел) ✅

Сейчас модерации контента фактически нет — только разовый CAS на джойне + слепое удаление медиа-типов. Это самая ценная фаза.

### 1.1 Модерация сообщений `[L]` ✅

Антиспам должен работать не только на входе.

- Фильтр сообщений от «новичков» (первые N сообщений / первые сутки): ссылки, форварды, @-меншены, telegram-инвайты.
- Обработка `edited_message` — повторная прогонка через фильтры (закрывает трюк «чистый текст → правка в спам»).
- Действие по срабатыванию настраиваемое: удалить / мьют / варн.

**Готово:** спам-ссылка от новичка удаляется; правка чистого сообщения в спам тоже ловится.

Зависит от: 2.3 (состояние «новизны» юзера удобно хранить в БД) — но MVP можно на in-memory.

### 1.2 Капча/верификация новых участников `[L]` ✅

- При входе — мьют + сообщение с кнопкой/простым challenge; прошёл за T секунд → размьют и приветствие, не прошёл → кик/бан.
- Конфиг таймаута и типа капчи.

**Готово:** автоматический бот без прохождения капчи не получает право писать.

Связано с: текущим `greet_chat_members` (капча встраивается в этот же флоу до приветствия) и CAS (CAS-хит → сразу бан, минуя капчу).

### 1.3 Управляемое удаление медиа `[M]` ✅

Сейчас голос/видео/кружки/локации сносятся у всех молча, включая админов.

- Вынести список запрещённых типов в конфиг (вкл/выкл по типу).
- Не трогать админов.
- Опционально — ответное уведомление автору («медиа этого типа запрещены»), с авто-удалением уведомления через N сек.

**Готово:** набор удаляемых типов управляется конфигом; админов не задевает; пользователь понимает причину.

### 1.4 Флуд-контроль `[M]` ✅

- Счётчик сообщений на юзера в окне времени; превышение → мьют на короткий срок + предупреждение.
- Пороги в конфиге.

**Готово:** серия из X сообщений за Y секунд приводит к временному мьюту.

Зависит от: 2.1 (временный мьют).

## Фаза 2 — Инструментарий модератора ✅

Расширение возможностей админа; опирается на состояние из Фазы 0–1.

### 2.1 Временные наказания и `/kick` `[M]` ✅

- `/mute 1h`, `/ban 1d` — парсинг длительности; снятие по `until_date` (нативный Telegram) либо по таймеру.
- `/kick` = ban + immediate unban.

**Готово:** `/mute 30m` возвращает права автоматически через 30 минут.

### 2.2 Система варнов `[L]` ✅

- `/warn` с причиной; счётчик на юзера; при достижении порога — авто-мьют/бан.
- `/warns`, `/unwarn` для просмотра и снятия.

**Готово:** N варнов автоматически наказывают; история варнов доступна.

Зависит от: 2.3 (БД).

### 2.3 Состояние / БД `[L]` ✅

- Хранилище (SQLite на старте, чтобы не тянуть инфраструктуру): варны, активные мьюты, «новизна» юзера, статистика.
- Слой репозитория, чтобы хендлеры не знали о деталях БД.

**Готово:** варны/мьюты/статистика переживают рестарт бота.

Является фундаментом для: 1.1, 2.2, частично 1.4.

### 2.4 Таргетинг по @username / ID `[M]` ✅

- Команды принимают `@username` или числовой id, а не только reply.
- Резолв username → id (с учётом ограничений Bot API; возможно через кэш увиденных юзеров в БД).

**Готово:** `/ban @user` и `/ban 123456` работают без наличия сообщения под рукой.

Зависит от: 2.3 (кэш username→id).

## Сводка зависимостей

```
Фаза 0  →  независимые быстрые правки (можно мерджить по одной)
Фаза 1  →  1.1 и 1.4 удобнее после 2.3, но MVP возможен на in-memory
Фаза 2  →  2.3 (БД) — фундамент для 1.1, 2.2, 2.4
```

Рекомендуемый порядок старта: вся Фаза 0 (закрывает баги и безопасность за день-два) → **2.3 БД** (разблокирует остальное) → Фаза 1 → остаток Фазы 2.

## Фаза 3 — Веб-панель администрирования ✅

Бот превращён в панель управления чатами: аналитика + журнал + управление из браузера (PR #97–#100).

### 3.1 Слой данных `[L]` ✅

- **Audit-log** — все действия модерации (ручные команды, автонаказания, капча, CAS, флуд, фильтры, вступления/выходы) пишутся в `audit_log` с актором, причиной и meta; событие записывается только после успешного действия в Telegram.
- **Мягкое удаление варнов** — `clear_warns`/`remove_last_warn` ставят `deleted_at` вместо физического удаления; история доступна для аналитики.
- **Подневная статистика сообщений** — агрегат `message_stats` (chat, day, user) вместо события на каждое сообщение.
- **Миграции схемы** — `PRAGMA user_version` + список миграций; v0-схема заморожена как baseline.
- **Ретеншен** — ежедневная asyncio-джоба чистит историю старше `storage.retention.days` (дефолт 365); живое состояние не трогается.

### 3.2 Дашборд (read-only) `[L]` ✅

- FastAPI + Jinja2 + htmx + Chart.js в одном процессе с ботом (общий event loop); при `web.enabled: false` поведение прежнее.
- Auth: Telegram Login Widget (HMAC-верификация), допуск по `admin_ids`, подписанная сессионная кука; dev-вход в DEBUG.
- Страницы: обзор чатов, аналитика чата (плитки, графики сообщений/вступлений/действий, топы), журнал с фильтрами и пагинацией.
- `/healthz` + Service/пробы в k8s.

### 3.3 Управление и экспорт `[M]` ✅

- Действия из панели: ban/unban/kick/mute/unmute по id или @username с длительностью и причиной; CSRF; всё аудируется с `meta.source=web`.
- Принудительный компактинг: dry-run предпросмотр → подтверждение → purge + VACUUM; аудируется.
- Экспорт: CSV (журнал с фильтрами, сообщения по дням, участники), `config.yaml`, `env.json` (чек-лист без значений секретов), консистентный бэкап БД (`VACUUM INTO`).

### 3.4 Anti-raid режим `[M]` ✅

- Всплеск вступлений (`moderation.raid`: порог/окно) включает усиленный режим на время (продлевается, пока волна не стихла): таймаут капчи вдвое короче, провал капчи = бан, нарушения новичков эскалируются до мьюта.
- Начало/конец рейда аудируются с параметрами и числом вступлений; предупреждение в чат; плитка состояния на дашборде.

### 3.5 Автобэкапы по расписанию `[M]` ✅

- Ежедневный `VACUUM INTO`-снапшот в `storage.backup.dir` с ротацией (хранятся последние `keep`); пропущенный за даунтайм прогон догоняется на старте.
- Опциональная выгрузка в S3-совместимое хранилище (минимальный SigV4-клиент на httpx, без boto3) с той же ротацией по префиксу; креды только из env.
- Ручной бэкап кнопкой на странице «Обслуживание» (CSRF, аудируется с актором); список локальных снапшотов там же.

## Что дальше

Согласованные кандидаты следующей итерации (в порядке выполнения):

- **UX панели `[M]`** — быстрые действия из строк журнала/топов, недельные/месячные агрегаты для длинных периодов, страница пользователя с его историей.

Закрыто попутно с Фазой 3: статистика для админа (веб-дашборд), экспорт журнала действий (CSV), миграции схемы БД, `/healthz`, измерение покрытия в CI (порог 85%).
